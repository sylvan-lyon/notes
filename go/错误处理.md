# Go 错误处理深度解析：从链到树，最佳实践与性能考量

## 1. 引言

Go 的错误处理机制在 1.13 版本引入了标准化的错误链（error chaining），通过 `Unwrap` 方法和 `errors.Is`/`As` 函数，使得错误的包装、传递和检查变得统一。Go 1.20 进一步支持多错误包装（`Unwrap() []error`），形成了错误树（error tree），能够表达多个并列的错误（例如通过 `errors.Join` 或 `fmt.Errorf` 的多个 `%w`）。理解这些底层机制对于编写健壮、可维护且高效的错误处理代码至关重要。

---

## 2. 底层实现

### 2.1 单错误包装：`Unwrap() error`

任何实现了 `Unwrap() error` 方法的类型都可以作为单错误包装器。例如 `fmt.Errorf` 使用 `%w` 时返回的内部类型 `*fmt.wrapError`（简化定义）：

```go
type wrapError struct {
    msg string
    err error
}

func (e *wrapError) Error() string { return e.msg }
func (e *wrapError) Unwrap() error { return e.err }
```

`errors.Unwrap` 函数简单地检查并调用该方法：

```go
func Unwrap(err error) error {
    // 这里有一个匿名的 interface { Unwrap() error }
    // 通过这个匿名借口查看 Unwrap 方法的情况
    // 这种方法没有官方命名，社区中叫它：
    // - 匿名接口类型断言
    // - 接口鸭子类型检查
    // - 内联接口断言
    u, ok := err.(interface { Unwrap() error })
    if !ok {
        return nil
    }
    return u.Unwrap()
}
```

### 2.2 多错误包装：`Unwrap() []error`

从 Go 1.20 开始，错误可以实现 `Unwrap() []error` 来包含多个子错误，形成树形结构。例如 `errors.Join` 返回的错误以及 `fmt.Errorf` 在多个 `%w` 时返回的内部类型 `*fmt.wrapErrors`：

```go
type wrapErrors struct {
    msg  string
    errs []error
}

func (w *wrapErrors) Error() string    { return w.msg }
func (w *wrapErrors) Unwrap() []error { return w.errs }
```

### 2.3 `errors.Is` 和 `errors.As` 的遍历逻辑

这两个函数能够递归遍历错误树，同时处理单错误和多错误包装。核心逻辑如下（伪代码）：

```go
func Is(err, target error) bool {
    for {
        if err == target { return true }
        // 依旧是上面提及的匿名类型断言
        if checker, ok := err.(interface{ Is(error) bool }); ok && checker.Is(target) {
            return true
        }
        switch u := err.(type) {
        case interface{ Unwrap() []error }:
            for _, e := range u.Unwrap() {
                if Is(e, target) {
                    return true
                }
            }
            return false
        case interface{ Unwrap() error }:
            err = u.Unwrap()
            if err == nil { return false }
        default:
            return false
        }
    }
}
```

`errors.As` 类似，但尝试将错误赋值给目标类型。

## 3. Error Chain 与 Error Tree

### 3.1 定义

- **错误链（Chain）**：每个错误只包装一个子错误，通过 `Unwrap() error` 形成线性链表。
- **错误树（Tree）**：一个错误可以包装多个子错误，通过 `Unwrap() []error` 形成分支结构。每个子错误又可以是链或树。

### 3.2 形成方式

- **单链**：`fmt.Errorf("... %w ...", err)`（单个 `%w`）或自定义类型实现 `Unwrap() error`。
- **树**：
  - `errors.Join(err1, err2, ...)` 返回的错误包含所有子错误。
  - `fmt.Errorf("... %w ... %w ...", err1, err2)`（多个 `%w`）返回实现 `Unwrap() []error` 的错误。
  - 自定义类型实现 `Unwrap() []error`。

### 3.3 遍历示例

假设有如下错误树：

```
root
├── child1 (实现了 Unwrap() error → grandchild)
└── child2 (无子错误)
```

调用 `errors.Is(root, target)` 会：
1. 检查 `root` 本身。
2. 调用 `root.Unwrap() []error`，得到 `[child1, child2]`。
3. 对 `child1` 递归：检查自身，然后解包得到 `grandchild` 并检查。
4. 对 `child2` 递归：直接检查。

最终遍历所有节点。

## 4. 两种错误处理模式：哨兵错误 vs 自定义错误类型

### 4.1 定义与示例

| 模式          | 定义                                  | 示例                                          |
| ----------- | ----------------------------------- | ------------------------------------------- |
| **哨兵错误**    | 预定义的包级错误变量，通过 `==` 或 `errors.Is` 比较 | `var ErrNotFound = errors.New("not found")` |
| **自定义错误类型** | 实现 `error` 接口的结构体，可携带额外信息           | `type NotFoundError struct{ ID int }`       |

### 4.2 优缺点对比

| 方面     | 哨兵错误                                            | 自定义错误类型                                                          |
| ------ | ----------------------------------------------- | ---------------------------------------------------------------- |
| **优点** | - 简单直观<br>- 性能高（全局变量，零分配）<br>- 与标准库一致<br>- 便于测试 | - 可携带结构化上下文<br>- 类型安全<br>- 可自定义 `Unwrap`/`Is`/`As`<br>- 适合复杂错误处理 |
| **缺点** | - 缺乏上下文信息<br>- 导致包间耦合（直接依赖变量）<br>- 难以扩展         | - 定义稍繁琐<br>- 每次返回可能分配新对象<br>- 调用方需了解具体类型<br>- 可能导致类型膨胀           |

### 4.3 适用场景

| 场景                 | 推荐模式                 |
| ------------------ | -------------------- |
| 简单错误信号（如 EOF、未找到）  | 哨兵错误                 |
| 基础库/标准库的错误         | 哨兵错误                 |
| 性能敏感路径             | 哨兵错误（避免分配）           |
| 需要传递结构化信息（错误码、字段名） | 自定义错误类型              |
| 多层包装需要保留上下文        | 自定义错误类型（或 `%w` + 哨兵） |
| API 边界返回丰富错误信息     | 自定义错误类型              |
| 复杂错误决策（根据字段值判断）    | 自定义错误类型              |

### 4.4 性能考量

| 维度 | 哨兵错误 | 自定义错误类型 |
|------|----------|----------------|
| **创建开销** | 无（全局变量） | 每次返回分配新对象（除非复用单例） |
| **比较开销** | `errors.Is` 指针比较，极快 | 类型断言 + 可能遍历树，稍慢 |
| **包装开销** | `%w` 分配小包装器 | 自定义类型可能已实现 Unwrap，仍需分配 |
| **内存占用** | 极小 | 取决于携带字段 |
| **遍历开销** | 线性链，快 | 树形结构可能遍历更多节点 |

**优化建议**：
- 热路径中优先使用哨兵错误。
- 自定义错误尽量不可变，如果可能复用单例（但通常需要动态数据）。
- 使用 `errors.Is`/`As` 而不是 `==`/类型断言，以支持包装。
- 避免过度包装，只在需要添加上下文时包装。

## 5. 最佳实践

1. **优先使用哨兵错误** 表示简单的错误类别，配合 `%w` 包装以添加上下文：
   ```go
   return fmt.Errorf("read config: %w", ErrNotFound)
   ```

2. **在 API 边界使用自定义错误类型**，提供结构化信息供上层处理：
   ```go
   type APIError struct {
       Code    int
       Message string
   }
   ```

3. **实现 `Unwrap` 方法**，使自定义错误可参与错误链。

4. **考虑实现 `Is` 和 `As` 方法**，自定义比较和转换逻辑，例如让一个错误匹配多个哨兵。

5. **避免暴露具体错误类型给所有调用方**：如果类型只在包内使用，保持未导出，包外通过 `errors.Is` 检查哨兵或通过接口方法获取信息。

6. **不要在性能关键循环中创建大量错误**，如果无法避免，使用预定义的哨兵错误。

7. **使用 `errors.As` 而不是类型断言**，以支持错误链遍历。

8. **合理使用多错误包装**：当需要聚合多个独立错误时（如校验错误列表），使用 `errors.Join` 或 `fmt.Errorf` 多 `%w`。

9. **文档化错误类型**：对于包公开的错误变量和类型，明确说明何时返回以及如何检查。

---

## 6. 总结

Go 的错误处理从简单的哨兵错误发展到支持链和树的包装机制，底层基于 `Unwrap` 方法族和标准库的遍历函数。理解单错误包装（链）和多错误包装（树）的实现，有助于设计出既灵活又高效的错误策略。哨兵错误适合简单信号和性能敏感场景，而自定义错误类型则提供了丰富的上下文和扩展性。在实际开发中，两者可以结合使用：用哨兵表示错误类别，用自定义类型携带细节，并通过 `%w` 包装保留可比较性。掌握这些底层原理和最佳实践，能让你在 Go 中写出更健壮、更易维护的错误处理代码。

# 从回溯到动态规划

## 回溯法引入

>[!NOTE]
>此处使用题目[打家劫舍](https://leetcode.cn/problems/house-robber)作为引子，介绍如何动态规划

### 分析

首先我们使用**回溯算法**进行解题，我们可以这样遍历每一种可能：
- 定义 `backtrack(i)` 函数表示讨论**从区间 `[i, len(nums)-1]` 中抢**的**最大值**
- 每层递归有两种可能：抢还是不抢
    - 不抢：第 `i` 个房子的下一个房子可以抢，于是调用 `backtrack(i + 1)` 讨论从 `[i + 1, len(nums)-1]` 中抢劫的最大值
    - 抢：由于抢了第 `i` 个房子，下一个房子不能抢，只能从第 `i + 2` 后面的房子里面抢，于是调用 `backtrack(i + 2)` 讨论从 `[i + 2, len(nums) - 1]` 中抢劫的最大值

### 代码实现

```go
func rob(nums []int) int {
    // 表示可以在 [i, len(num)-1] 中抢
    var backtrack func(int) int
        backtrack = func(i int) int {
        if i >= len(nums) {
            return 0
        }
        // 不抢这一家, 之后可以在 [i+1, len(nums)-1] 中抢
        // 抢这一家, 之后可以在 [i+2, len(nums)-1]中抢 (跳过下一家)
        return max(backtrack(i + 1), backtrack(i + 2) + nums[i])
    }
    return backtrack(0)
}
```

## 反思

我们兴高采烈地完成上述代码，以为自己又战胜了一道题目时，冷水来了：超时

- 时间复杂度：我们可以**粗略地估计**，对于每一个房子有两种可能，抢还是不抢，于是**节点个数是 $O(2^n)$ ，实际上会更少**，因为有相邻家舍不能同时打劫的限制。每次讨论一个单节点需要的时间是 $O(1)$ 的，于是**时间复杂度近似 $O(2^n)$**
- 空间复杂度：假设我们每一家都不抢，**调用栈最终会是 $O(n)$ 的长度**，由于没有使用额外的堆内存，所以空间复杂度近似 $O(n)$

## 记忆化搜索

### 分析

观察回溯树，我们可以发现**不少节点被重复讨论**，而从 `[i, len(nums)-1]` 的区间中抢劫所得的最大值是固定的。于是我们可以想到用一个 `cache` 来存储相应的最佳方案。

>[!NOTE]
>其实我认为这也算是一种剪枝

### 代码实现

#### 用哈希表

```go
func rob(nums []int) int {
    cache := make(map[int]int, len(nums))
    // 表示可以在 [i, len(num)-1] 中抢
    var backtrack func(int) int
    backtrack = func(i int) int {
        if i >= len(nums) {
            return 0
        }
        // 不抢这一家, 之后可以在 [i+1, len(nums)-1] 中抢
        // 抢这一家, 之后可以在 [i+2, len(nums)-1]中抢 (跳过下一家)
        maxA, okA := cache[i + 1]
        maxB, okB := cache[i + 2]
        if !okA {
            cache[i + 1] = backtrack(i + 1)
            maxA = cache[i + 1]
        }
        if !okB {
            cache[i + 2] = backtrack(i + 2)
            maxB = cache[i + 2]
        }
        return max(maxA, maxB + nums[i])
    }
    return backtrack(0)
}
```

#### 用缓存数组

```go
func rob(nums []int) int {
    cache := make([]int, len(nums))
    // 初始化为值域以外的值
    for i := range cache {
        cache[i] = -1
    }
    // 表示可以在 [i, len(num)-1] 中抢
    var backtrack func(int) int
    backtrack = func(i int) int {
        if i >= len(nums) {
            return 0
        }
        // 如果当前节点已经被计算过
        if cache[i] != -1 {
            return cache[i]
        }
        // 不抢这一家, 之后可以在 [i+1, len(nums)-1] 中抢
        // 抢这一家, 之后可以在 [i+2, len(nums)-1]中抢 (跳过下一家)
        maxA := backtrack(i + 1)
        maxB := backtrack(i + 2) + nums[i]
        cache[i] = max(maxA, maxB)
        return cache[i]
    }
    return backtrack(0)
}
```

### 时空复杂度分析

- 时间复杂度：由于现在**每个节点只会被计算一遍**，剩下的都是从 `cache` 中以 $O(1)$ 的时间复杂度获取最佳解，所以时间复杂度为 $O(n)$
- 空间复杂度：调用栈最大长度仍然是 $O(n)$ 的，还有一个**长度为 `O(n)` 的缓存**，所以空间复杂度是 $O(n)$ 的

## 优化

### 分析

上面的代码其实可以写成这样的公式，记 $f(i) = backtrack(i)$，$n = len(nums)$：

$$
\begin{align}
&f(i) = \max(f(i+1), f(i+2)+\text{nums[i]}) \ , \quad i\in[0,n-1] \\
&f(n)=0 \\
&f(n+1)=0
\end{align}
$$

也就是说前一个依赖于后两个的结果，这是一个倒推公式。于是

### 数组实现

我们用一个叫 $f$ 的数组表示上面的函数 $f$

```go
func rob(nums []int) int {
    n := len(nums)
    f := make([]int, n + 2)
    f[n] = 0
    f[n + 1] = 0      
    for i := n - 1; i >= 0; i -= 1 {
        f[i] = max(f[i+1], f[i+2] + nums[i])
    }
    return f[0]
}
```


### 两个变量实现

观察上面的数组，我们可以发现在 `f[i]` 算出来后，`f[i+2]` 就再也没使用过，于是我们可以用类似斐波那契的方式：

```go
func rob(nums []int) int {
    n := len(nums)
    // 由于变量名中不能有 '+', 所以如此表示, 可不是减号啊
    // f -> f(i), fi_1 -> f(i+1), fi_2 -> f(i+2)
    f, fi_1, fi_2 := 0, 0, 0
    for i := n - 1; i >= 0; i -= 1 {
        f = max(fi_1, fi_2 + nums[i])
        fi_2 = fi_1
        fi_1 = f
    }
    return f
}
```

### 时空复杂度分析

- 时间复杂度：很明显，是 $O(n)$ 的
- 空间复杂度：数组实现用了一个 $O(n)$ 的数组，而两个变量实现的只用了两个额外的变量，空间复杂度 $O(1)$

## 推广

### 从回溯到动态规划

我们刚刚走过的这条路，正是算法设计中从 **“暴力搜索”** 到 **“动态规划”** 的经典进化过程。为了能在未来面对更复杂的问题时也能游刃有余，我们来把刚才经历的每一步，都用动态规划领域的“行话”重新解读一遍。

#### 1. 最优子结构（Optimal Substructure）

在打家劫舍问题中，我们最终要计算的是 **“从全部房子中能抢到的最大金额”**。  
当我们写出递推式  

$$
f(i) = \max\big( f(i+1),\ f(i+2) + \text{nums}[i] \big)
$$

时，实际上就揭示了该问题的 **最优子结构** 性质：

> **一个全局最优解包含其子问题的最优解。**

具体来说，对于**区间 $[i, n-1]$ 的最优解 $f(i)$**，要么**来自 $[i+1, n-1]$ 的最优解（不抢 $i$）**，要么**来自 $[i+2, n-1]$ 的最优解加上 $\text{nums}[i]$（抢 $i$）**。

>[!IMPORTANT]
> 也就是说，原问题的最优决策，完全由它的“子区间”的最优决策决定，而不会出现**“子问题不是最优，但拼起来反而更好”**的悖论。  
> 
>这一性质是动态规划能够正确推导的基础。

#### 2. 重叠子问题（Overlapping Subproblems）

在最初的回溯代码中，我们为什么超时？因为递归树里充斥着**大量重复计算**。比如，$f(2)$ 可能既被 $f(0)$ 的**“不抢分支”**调用，又被 $f(1)$ 的**“抢分支”**调用。

这些反复出现的子问题，就是 **重叠子问题**。  

动态规划正是利用了这一点，通过记忆化（缓存）或者自底向上的填表法，让每个子问题只求解一次，从而将指数级复杂度降为多项式级。

#### 3. 状态与状态转移方程

- **状态**：我们定义 $f(i)$ 为 **“从第 $i$ 间房子开始往后抢，所能获得的最大金额”**。  
  在动态规划中，状态就是“子问题”的形式化描述，它必须能完整刻画一个决策阶段。
- **状态转移方程**：上面那个递推式  
  $f(i) = \max(f(i+1), f(i+2) + \text{nums}[i])$  
  就是 **状态转移方程**。它描述了状态之间如何递推，是动态规划的核心公式。

#### 4. 自顶向下 vs 自底向上

- **自顶向下（Top-Down）**：就是我们一开始写的 **记忆化搜索**。从大问题出发，递归求解，遇到重复子问题时直接查缓存。它保留了回溯的直观性，但加入了记忆化。
- **自底向上（Bottom-Up）**：就是最后优化出的数组迭代法。我们从最小的子问题（$f(n)$、$f(n+1)$）开始，逐步递推出更大的 $f(i)$，最终得到 $f(0)$。这种写法没有递归调用，通常效率更高，空间也可以进一步压缩。

两种方式本质等价，但自底向上往往更简洁，也更容易进行空间优化。

#### 5. 动态规划的“三要素”

几乎所有动态规划问题都可以从以下三个角度入手：

1. **状态定义** —— 明确 $dp[i]$ 或 $dp[i][j]$ 表示什么。
2. **转移方程** —— 如何通过更小规模的状态推导当前状态。
3. **初始条件与边界** —— 最小子问题的答案（比如 $f(n)=0$）。

在打家劫舍中，我们的状态定义、转移方程、边界条件都非常清晰，这正是它作为动态规划入门题的魅力所在。

### 从“打家劫舍”到更广阔的动态规划世界

当你掌握了上述概念后，可以尝试用同样的思路去分析其他经典问题，比如：

- **爬楼梯**（LeetCode 70）：状态 $dp[i] = dp[i-1] + dp[i-2]$，本质是斐波那契。
- **不同路径**（LeetCode 62）：状态 $dp[i][j] = dp[i-1][j] + dp[i][j-1]$，是二维递推。
- **背包问题**：状态 $dp[i][w]$ 表示考虑前 $i$ 个物品、容量为 $w$ 时的最大价值，转移方程涉及“选”与“不选”。

你会发现，它们都遵循着相同的模式：**找出重叠子问题 → 定义状态 → 写出状态转移方程 → 确定边界 → 选择实现方式（自顶向下或自底向上）**。

### 最后的叮嘱

回溯是天然的思维起点，它模拟了人类穷举所有可能的过程；而动态规划则是利用问题结构进行优化的艺术。当你感到一道题可以用回溯写出，却超时的时候，不妨问自己两个问题：

- 是否存在重复计算的重叠子问题？
- 能否用某个“状态”来描述子问题，并写出递推关系？

如果答案都是“是”，那么恭喜你，你已经站在了动态规划的大门前。
